## Project overview

Nonogram (picross) web application. Nonograms can be drawn manually cell by cell in the browser (row/column clues computed automatically from the drawing) or imported from an external source. No CLI — a web UI only, with a button to send the resulting PDF straight to a reMarkable tablet.

**Stack:** Node.js / TypeScript, npm workspaces (monorepo), Fastify, Vitest, Biome
**Type:** Full-stack (web API/server + shared core, no CLI)

## Language

## Architecture

npm workspaces monorepo with a `core` package holding the nonogram domain logic (grid, clue computation, PDF/SVG rendering) and the reMarkable Cloud integration (auth, upload), consumed by the `web` package (Fastify backend + static frontend). No CLI package — creation is manual, through the browser only.

```
remarkable-nonogram-generator/
├── packages/
│   ├── core/         # nonogram domain (grid, clues, rendering — TODO) + reMarkable Cloud client (auth, upload, credential store)
│   └── web/           # Fastify backend + static frontend (public/index.html, app.js, style.css), consuming @remarkable-nonogram-generator/core
├── tsconfig.base.json # TypeScript config shared by all packages
└── vitest.config.ts   # root test config (covers all packages)
```

Current state: the reMarkable pairing/connection flow (`authenticate`, `uploadPdf`, file-based credential store, `/api/remarkable/status` and `/api/remarkable/pair` routes) is implemented and tested, ported from the sibling project `remarkable-maze-generator` since it is fully generic. The nonogram domain itself (grid data model, manual cell editor UI, clue computation, SVG/PDF rendering, `/api/nonograms/*` routes, "send to reMarkable" wiring) is **not implemented yet** — build it with `/vibe:feature`, one slice at a time, following the TDD workflow below.

## Testing conventions

- Test location: `packages/*/src/**/*.test.ts` (a test file next to the module it covers)
- Test runner: Vitest (`npm test` at the root runs every package)
- One test file per source module
- Test names must describe behavior: `"returns an empty grid when size is 0"`, not `"it works"`
- Always cover: happy path + at least 2 edge cases + error/invalid input
- The `core` package has no dependency on any browser-specific runtime (no DOM) so it stays testable in isolation and shareable if another frontend is ever added
