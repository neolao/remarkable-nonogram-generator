# Module: nonogram-url-import
**Role:** Reconstructs a nonogram grid from a nonograms.org puzzle URL, instead of a manual drawing, by rendering the page through a headless browser and reading its clue numbers directly as HTML text (no OCR — see ADR `011`), then running a shared logic-only solver over those clues. Never trusts the source for the actual cell layout, only for clues, which are fed into the solver (see [[nonogram-domain]] for the "clues always derived from cells" rule this satisfies, and ADRs `011-nonograms-org-url-import-via-headless-browser`, `012-remove-image-based-ocr-import`). This used to be one of two import paths (the other being an OCR-based image upload, ADR `010`); the image upload path was removed (ADR `012`, backlog item 024) once the URL path proved more accurate, and backlog item 023 — which existed only to improve the removed OCR path's accuracy — was closed as moot alongside it.
**Files:** `packages/core/src/domain/nonogram-line-solver.ts`, `packages/core/src/infrastructure/nonogram-page-clues.ts`, `packages/core/src/application/nonogram-url-import.ts`
**Exports:**
- `solveNonogramFromClues(width, height, clues: NonogramClues): Nonogram` — pure logical nonogram solver, in two stages. First, it iteratively narrows each row/column (given its clue and any cells already fixed by crossing lines) via a leftmost/rightmost "push" line solver: each line's clue-blocks are pushed as far left, then as far right, as the already-known cells allow, and any cell landing in the same block in both extremes is forced to that state — no placement list is ever materialized (see ADR `018`, replacing the original enumeration-based approach of ADR `010`). This repeats until no further deduction is possible. If cells remain undetermined after that, it does bounded single-level hypothesis probing: each still-undetermined cell is tested filled and empty by cloning the grid and re-running the same propagation; a state that leads to a line with no valid placement is a contradiction, so the cell must be the other state. Rounds repeat (committing and re-propagating any deductions found) until a round makes no progress. This never guesses at random and never explores combinations of multiple cells' hypotheses together (no recursive backtracking), so it stays bounded (at most `width * height` rounds) and stops at a "requires guessing" error for puzzles needing deeper multi-cell reasoning, exactly as it did for puzzles needing any hypothesis-testing before this was added — see the fix that reversed ADR `010`'s original "never backtracks/guesses" scope after real nonograms.org puzzles turned out to need this single level of hypothesis-testing. Throws if a line's clue is impossible for its length, if `clues.rowClues`/`columnClues` don't match `height`/`width`, or if the grid still can't be fully determined this way.
- `extractCluesFromPage(url, pageRenderer?): Promise<{ width, height, clues: NonogramClues }>` — validates the URL matches a nonograms.org puzzle URL pattern, renders the page (default: real headless browser), reads the puzzle's `Size: WxH` text and its clue-number cells (`nmv{col}_{slot}` for columns; `nmh{slot}_{row}` for rows — note the inconsistent id ordering between the two, confirmed by direct inspection, not a typo), groups them by row/column and sorts by slot to get each line's clue in the right order, defaulting an entirely blank line to `[0]`.
- `PageRenderer` — `{ renderNonogramPage(url): Promise<RawNonogramPageData> }`, injectable (defaults to `createPuppeteerPageRenderer()`) so tests can substitute a fake instead of running a real headless browser, per the project's established dependency-injection pattern.
- `createPuppeteerPageRenderer(): PageRenderer` — the real default renderer (Puppeteer, headless Chromium): loads the puzzle page, confirms `#nonogram_table` exists, then reads size text and clue cells via `page.evaluate()`. Its evaluate callbacks avoid any named/assigned inner function (inlined loops instead) because esbuild's `__name()`-keeping transform (used by `tsx`/Vite/Vitest) otherwise leaks an undefined reference into the callback once it's serialized and re-run inside Puppeteer's isolated browser context — a real bug hit and fixed during this feature, not a style choice.
- `importNonogramFromUrl(url, { pageRenderer? }): Promise<Nonogram>` — chains `extractCluesFromPage` into `solveNonogramFromClues`.
**Depends on:** `modules/nonogram-domain.md` (`createNonogram`, `computeNonogramClues`'s `NonogramClues` type). Within `core`, `nonogram-line-solver.ts` (pure) sits in `domain/`, the Puppeteer-based `nonogram-page-clues.ts` sits in `infrastructure/`, and `nonogram-url-import.ts` (the use case chaining them) sits in `application/` (see ADR `016-ddd-layered-core-package-structure`).

Notes:
- The URL import path is a personal-use decision made with the CGU conflict explicitly acknowledged, not resolved — see ADR `011` and `.vibe/backlog/done/020-import-nonograms-from-nonograms-org.md`.
- A Puppeteer-based render can take close to (occasionally over) 30s on a cold Chromium start; the web server's connection timeout was raised from 30s to 60s to accommodate this (see [[web-server]]).
- The OCR-based image upload path (`tesseract.js`/`pngjs`, an `extractCluesFromImage`/`importNonogramFromImage` pair) was removed (ADR `012`) after real-world testing showed it was measurably less reliable than reading clue text directly; `solveNonogramFromClues` and the general "never trust the source's cells" rule are unchanged and still shared by any future import path.
- Previously unresolved limitation, now fixed (ADR `018`, backlog item `030`): the original enumeration-based line solver could blow up to hundreds of millions of array allocations for a single long, densely-clued line (real-world-verified against nonograms.org puzzle `80350`, a 45x45 grid — one line alone had 129,024,480 valid placements), exhausting the Node.js heap and crashing the whole server process during the very first propagation pass, before any hypothesis probing. The push-based replacement solves that same line in under 1ms.
