# Module: docker-deployment
**Role:** Packages the whole app (both workspaces) as a single Docker image for one-command deployment, notably on a Synology NAS, with saved nonograms and reMarkable Cloud credentials persisted via a mounted volume.
**Files:** `Dockerfile`, `docker-compose.yml`, `.dockerignore`, `docs/docker.md`
**Exports:** n/a (infrastructure, not application code)
**Depends on:** `modules/web-server.md` (runs `packages/web`'s server), `modules/nonogram-persistence.md` and `modules/remarkable-cloud.md` (both already default their on-disk storage to `homedir()/.config/remarkable-nonogram-generator/...`, so mounting a volume at `/home/node/.config/remarkable-nonogram-generator` — the container's `node` user's home — persists both without any application code change)

## Notes
- Multi-stage build: a `build` stage runs the TypeScript build (`npm run build`) purely as a type-check gate — its `dist` output is not what ships. The runtime stage actually starts the server via `tsx` on the TypeScript source (`npm start` → `tsx src/server.ts` in `packages/web`), the same mechanism already used by local dev (`npm run web` → `tsx watch`), because `core`'s `package.json` (`main`/`types`/`exports`) deliberately points at its own TypeScript source rather than a built `dist` — see [ADR 013](decisions/013-docker-runs-server-via-tsx-not-compiled-dist.md). As a consequence, `tsx` moved from `packages/web`'s `devDependencies` to `dependencies`, and `packages/web`'s `start` script now runs `tsx src/server.ts` instead of the previously broken (`ERR_UNKNOWN_FILE_EXTENSION`) `node dist/server.js`.
- The runtime image installs Chromium via the Debian `chromium` apt package (`PUPPETEER_EXECUTABLE_PATH`, `PUPPETEER_SKIP_DOWNLOAD=true`) instead of Puppeteer's own bundled download, so the same Dockerfile produces a working multi-arch (`amd64`/`arm64`) image — needed since Puppeteer only ships prebuilt Chromium for `linux x64` and Synology NAS models are Intel/AMD or ARM depending on the model — see [ADR 014](decisions/014-docker-uses-system-chromium-not-puppeteer-bundled.md). This is what backs the URL-import feature ([[nonogram-url-import]]) once containerized.
- The listen port reuses the existing `PORT` environment variable ([[web-server]]) — no Docker-specific configuration needed beyond mapping it.
