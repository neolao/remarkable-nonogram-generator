# Module: nonogram-rendering
**Role:** Renders a nonogram grid and its computed clues to an image or document format — SVG for live preview in the web UI, PDF for download/printing and sending to a reMarkable device. Pure rendering, no persistence or HTTP concerns.
**Files:** `packages/core/src/nonogram-svg.ts`, `packages/core/src/nonogram-pdf.ts`
**Exports:**
- `renderNonogramToSvg(nonogram, options?): string` — renders a `Nonogram` to an SVG string: one rect per grid cell (filled cells black, empty cells white), row clues right-aligned in a left margin, column clues bottom-aligned in a top margin, sized from `computeNonogramClues`. Clue numbers stack in a margin slot narrower than a full grid cell (75% of it) so the margin takes less of the page while numbers keep their full size — the alignment axis (row height / column width) still matches the grid's cell size exactly (see ADR `009-clue-margin-slot-smaller-than-grid-cell`). A thicker gridline is drawn every 5 interior rows/columns (indices 5, 10, …, strictly inside the grid — the outer border is never thickened, even when the total width/height is itself a multiple of 5). Throws if `options.cellSizePx` is not a positive number.
- `RenderNonogramToSvgOptions` — `{ cellSizePx?: number }` read-only options type
- `renderNonogramToPdf(nonogram, options?: RenderNonogramToPdfOptions): Promise<Uint8Array>` — renders a `Nonogram` to a PDF sized to the reMarkable 2 page (`REMARKABLE_2_PAGE_WIDTH_PT` x `REMARKABLE_2_PAGE_HEIGHT_PT`), mirroring the SVG renderer's grid/clue layout — including the same narrower clue-margin-slot packing — but with a cell size dynamically shrunk to fit the fixed page (same approach as the sibling maze project's `maze-pdf.ts`), guaranteeing the largest grid allowed by the domain model still fits without clipping; the page-fitting computation weights margin cells by the same narrower ratio, so the space freed by tighter clue packing grows the puzzle grid's cell size instead of going unused. Same thicker-every-5-interior-lines convention as the SVG renderer. Page 1 always renders every cell empty/white regardless of `nonogram.cells` — the blank puzzle to solve on the device; `cells` is only used to compute the clues there. When `options.includeSolution` is true, a second page is appended reusing the same layout/clue drawing but with cells rendered filled (black) or empty per `nonogram.cells`, matching the SVG preview's convention — the solved grid to check the answer against. Throws if the nonogram's `cells` don't match its declared dimensions.
- `RenderNonogramToPdfOptions` — `{ includeSolution?: boolean }` read-only options type
- `REMARKABLE_2_PAGE_WIDTH_PT`, `REMARKABLE_2_PAGE_HEIGHT_PT` — reMarkable 2 page dimensions in PDF points, derived from the domain model's pixel constants at 226 DPI
**Depends on:** `modules/nonogram-domain.md` (consumes `Nonogram` and `computeNonogramClues`); `nonogram-pdf.ts` uses the `pdf-lib` package (see ADR `003-pdf-lib-for-nonogram-pdf-rendering`)
