# Module: remarkable-cloud
**Role:** reMarkable Cloud integration — device pairing/authentication, credential persistence, PDF upload, and the "send a saved nonogram" use case. The core layer (`remarkable-auth.ts`, `remarkable-upload.ts`, `remarkable-credential-store.ts`, `send-nonogram.ts`, `network-timeout.ts`) is fully generic/domain-level (no HTTP concerns); `packages/web/src/remarkable-routes.ts` is a thin HTTP-mapping layer on top of it.
**Files:** `packages/core/src/infrastructure/remarkable-auth.ts`, `packages/core/src/infrastructure/remarkable-upload.ts`, `packages/core/src/application/remarkable-credential-store.ts`, `packages/core/src/application/send-nonogram.ts`, `packages/core/src/infrastructure/network-timeout.ts`, `packages/web/src/file-credential-store.ts`, `packages/web/src/remarkable-routes.ts`
**Exports:**
- `authenticate(store, pairingCode, options?, client?): Promise<RemarkableSession>` — pairs a new device or reuses stored credentials; `client` (default: the real `{ register, remarkable }` from `rmapi-js`) can be injected for testing without mocking the `rmapi-js` module (dependency-inversion: mirrors how `CredentialStore` is already injected)
- `RemarkableClient` — `{ register, remarkable }` injectable client shape, defaults to the real `rmapi-js` functions
- `uploadPdf(session, pdfBytes: Uint8Array, visibleName, options?): Promise<void>` — uploads PDF bytes directly to the account root or a named folder (no filesystem access — callers render/hold the bytes themselves; this replaced an earlier `filePath` + `readFile` callback design that forced non-file callers to fake a path)
- `withTimeout(promise, timeoutMs, message): Promise<T>` — races a promise against a timer, rejecting with `message` if the timer wins first; every `rmapi-js` network call in `remarkable-auth.ts`/`remarkable-upload.ts` (register, remarkable, listIds, getMetadata, putPdf, uploadPdf) is wrapped in this with a 30s timeout, since `rmapi-js` accepts no timeout/abort option itself. Inside `uploadPdf`'s folder-lookup walk, a single entry's `getMetadata` call is retried once after a `withTimeout` timeout before the whole lookup gives up on it — a lone slow entry among the potentially hundreds an account can hold shouldn't abort the entire "send to folder" flow
- `sendNonogramToRemarkable(nonogramStore, credentialStore, id, options?): Promise<SendNonogramResult>` — the full "send a saved nonogram" use case, independent of Fastify: loads the nonogram (`not_found` outcome if unknown), checks for stored credentials (`not_authenticated` outcome if none), re-authenticates (`auth_failed` outcome with the error message on failure), renders it to PDF via `renderNonogramToPdf` ([[nonogram-rendering]]) honoring `options.includeSolution`, and uploads it under the saved nonogram's name honoring `options.folder` (`upload_failed` outcome with the error message on failure, `sent` outcome with `visibleName` on success)
- `SendNonogramOptions` — `{ folder?, includeSolution? }`; `SendNonogramResult` — discriminated union over the outcomes above
- `createFileCredentialStore(filePath): CredentialStore` — persists the device token to a `0o600` JSON file (default: `~/.config/remarkable-nonogram-generator/credentials.json`)
- `registerRemarkableRoutes(app, credentialStore, nonogramStore)` — registers `GET /api/remarkable/status`, `POST /api/remarkable/pair`, and `POST /api/nonograms/:id/send` on a Fastify instance. The send route calls `sendNonogramToRemarkable` and maps its outcome to HTTP: `not_found` → 404, `not_authenticated` → 409 `{ error: "not_authenticated" }`, `auth_failed`/`upload_failed` → 502 with the outcome's message, `sent` → 200 with `visibleName`
**Depends on:** `rmapi-js` (reMarkable Cloud client); `send-nonogram.ts` additionally depends on `modules/nonogram-persistence.md` (via `NonogramStore`) and `modules/nonogram-rendering.md`; `packages/web/src/remarkable-routes.ts` now only depends on this module (no direct dependency on rendering/persistence anymore — that moved into `send-nonogram.ts`). Within `core`, `remarkable-auth.ts`/`remarkable-upload.ts`/`network-timeout.ts` sit in `infrastructure/`; the `CredentialStore` port and the `sendNonogramToRemarkable` use case sit in `application/` (see ADR `016-ddd-layered-core-package-structure`).
