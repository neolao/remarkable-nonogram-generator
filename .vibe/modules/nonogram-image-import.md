# Module: nonogram-image-import
**Role:** Reconstructs a nonogram grid from an image of an external puzzle source, instead of a manual drawing — for the first supported type ("nonograms.org"), an image of the site's printable puzzle page (a blank grid with row/column clue numbers in the margins, no filled cells to sample). Since there are no cells to read directly, the pipeline reads the clue numbers via OCR, then reconstructs the grid with a logic-only nonogram solver — never trusting OCR'd numbers as the final clues, only as an intermediate input (see [[nonogram-domain]] for the "clues always derived from cells" rule this satisfies, and ADR `010-nonograms-org-image-import-via-ocr-and-solver`).
**Files:** `packages/core/src/nonogram-line-solver.ts`, `packages/core/src/nonogram-clue-ocr.ts`, `packages/core/src/nonogram-image-import.ts`
**Exports:**
- `solveNonogramFromClues(width, height, clues: NonogramClues): Nonogram` — pure logical nonogram solver: iteratively narrows each row/column (given its clue and any cells already fixed by crossing lines) by enumerating the placements of that line's clue-blocks still consistent with what's known, repeating until no further deduction is possible. Throws if a line's clue is impossible for its length, if `clues.rowClues`/`columnClues` don't match `height`/`width`, or if the grid can't be fully determined this way (deliberately never backtracks/guesses — see ADR `010`, since nonograms.org guarantees every puzzle is logically solvable without guessing).
- `extractCluesFromImage(imageBuffer, { width, height, ocrEngine? }): Promise<NonogramClues>` — decodes a PNG, locates the grid's own bordering gridlines (the last `width+1`/`height+1` near-fully-dark rows/columns found — everything before them is the image's outer frame/clue margin, not the grid), then reads each row's/column's clue numbers one grid-aligned "slot" at a time (slots tile the margin's actual pixel span, right-aligned toward the grid, since a line needing fewer numbers than the max leaves its outermost slots blank) via OCR, defaulting a line with no recognized slots to the empty-line clue `[0]`. Throws if the image isn't a valid PNG, if `width`/`height` aren't positive integers, or if the declared grid size doesn't match enough detected gridlines.
- `OcrEngine` — `{ recognizeDigits(imageRegion): Promise<string>, terminate(): Promise<void> }`, injectable (defaults to `createTesseractOcrEngine()`) so tests can substitute a fake instead of running real OCR, per the project's established dependency-injection pattern.
- `createTesseractOcrEngine(): OcrEngine` — the real default engine (`tesseract.js`, PSM `SINGLE_LINE` restricted to digits): each slot crop is upscaled 4x before recognition, measured to meaningfully improve accuracy on glyphs this small; a single worker is created lazily and reused across all slots in one `extractCluesFromImage` call.
- `SupportedImageImportType` — currently just `"nonograms.org"`; the type string is a plain dispatch key so more types/parsers can be added later without touching this one.
- `importNonogramFromImage(imageType, imageBuffer, { width, height, ocrEngine? }): Promise<Nonogram>` — dispatches on `imageType` (throws `Unsupported image import type: ...` for anything but `"nonograms.org"`), then chains `extractCluesFromImage` into `solveNonogramFromClues`.
**Depends on:** `modules/nonogram-domain.md` (`createNonogram`, `computeNonogramClues`'s `NonogramClues` type)

Note: OCR accuracy on very small clue-number glyphs is imperfect even after upscaling (a few digits per puzzle can still be misread) — this is an accepted limitation, not a bug: `solveNonogramFromClues`'s strict all-or-nothing validation means misread clues overwhelmingly surface as a clear rejection (impossible or unsolvable-by-logic error) rather than a silently wrong puzzle, since a full-grid solution needs every line's clue to be simultaneously consistent.
