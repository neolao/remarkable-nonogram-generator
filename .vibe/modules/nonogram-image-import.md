# Module: nonogram-image-import
**Role:** Reconstructs a nonogram grid from an external puzzle source, instead of a manual drawing, via two paths that share the same logic-only solver: (1) an image of nonograms.org's printable puzzle page (a blank grid with row/column clue numbers in the margins, no filled cells to sample) — clue numbers are read via OCR; (2) a nonograms.org puzzle URL rendered through a headless browser, reading the same clue numbers directly as HTML text from the page's interactive widget (no OCR needed, and empirically more accurate — see ADR `011`). Neither path ever trusts the source for the actual cell layout, only for clues, which are fed into the solver (see [[nonogram-domain]] for the "clues always derived from cells" rule this satisfies, and ADRs `010-nonograms-org-image-import-via-ocr-and-solver`, `011-nonograms-org-url-import-via-headless-browser`).
**Files:** `packages/core/src/nonogram-line-solver.ts`, `packages/core/src/nonogram-clue-ocr.ts`, `packages/core/src/nonogram-image-import.ts`, `packages/core/src/nonogram-page-clues.ts`, `packages/core/src/nonogram-url-import.ts`
**Exports:**
- `solveNonogramFromClues(width, height, clues: NonogramClues): Nonogram` — pure logical nonogram solver: iteratively narrows each row/column (given its clue and any cells already fixed by crossing lines) by enumerating the placements of that line's clue-blocks still consistent with what's known, repeating until no further deduction is possible. Throws if a line's clue is impossible for its length, if `clues.rowClues`/`columnClues` don't match `height`/`width`, or if the grid can't be fully determined this way (deliberately never backtracks/guesses — see ADR `010`, since nonograms.org guarantees every puzzle is logically solvable without guessing). Shared unchanged by both import paths below.
- `extractCluesFromImage(imageBuffer, { width, height, ocrEngine? }): Promise<NonogramClues>` — decodes a PNG, locates the grid's own bordering gridlines (the last `width+1`/`height+1` near-fully-dark rows/columns found — everything before them is the image's outer frame/clue margin, not the grid), then reads each row's/column's clue numbers one grid-aligned "slot" at a time (slots tile the margin's actual pixel span, right-aligned toward the grid, since a line needing fewer numbers than the max leaves its outermost slots blank) via OCR, defaulting a line with no recognized slots to the empty-line clue `[0]`. Throws if the image isn't a valid PNG, if `width`/`height` aren't positive integers, or if the declared grid size doesn't match enough detected gridlines.
- `OcrEngine` — `{ recognizeDigits(imageRegion): Promise<string>, terminate(): Promise<void> }`, injectable (defaults to `createTesseractOcrEngine()`) so tests can substitute a fake instead of running real OCR, per the project's established dependency-injection pattern.
- `createTesseractOcrEngine(): OcrEngine` — the real default engine (`tesseract.js`, PSM `SINGLE_LINE` restricted to digits): each slot crop is upscaled 4x before recognition, measured to meaningfully improve accuracy on glyphs this small; a single worker is created lazily and reused across all slots in one `extractCluesFromImage` call.
- `SupportedImageImportType` — currently just `"nonograms.org"`; the type string is a plain dispatch key so more types/parsers can be added later without touching this one.
- `importNonogramFromImage(imageType, imageBuffer, { width, height, ocrEngine? }): Promise<Nonogram>` — dispatches on `imageType` (throws `Unsupported image import type: ...` for anything but `"nonograms.org"`), then chains `extractCluesFromImage` into `solveNonogramFromClues`.
- `extractCluesFromPage(url, pageRenderer?): Promise<{ width, height, clues: NonogramClues }>` — validates the URL matches a nonograms.org puzzle URL pattern, renders the page (default: real headless browser), reads the puzzle's `Size: WxH` text and its clue-number cells (`nmv{col}_{slot}` for columns; `nmh{slot}_{row}` for rows — note the inconsistent id ordering between the two, confirmed by direct inspection, not a typo), groups them by row/column and sorts by slot to get each line's clue in the right order, defaulting an entirely blank line to `[0]`.
- `PageRenderer` — `{ renderNonogramPage(url): Promise<RawNonogramPageData> }`, injectable (defaults to `createPuppeteerPageRenderer()`) for the same fake-in-tests/real-by-default reason as `OcrEngine`.
- `createPuppeteerPageRenderer(): PageRenderer` — the real default renderer (Puppeteer, headless Chromium): loads the puzzle page, confirms `#nonogram_table` exists, then reads size text and clue cells via `page.evaluate()`. Its evaluate callbacks avoid any named/assigned inner function (inlined loops instead) because esbuild's `__name()`-keeping transform (used by `tsx`/Vite/Vitest) otherwise leaks an undefined reference into the callback once it's serialized and re-run inside Puppeteer's isolated browser context — a real bug hit and fixed during this feature, not a style choice.
- `importNonogramFromUrl(url, { pageRenderer? }): Promise<Nonogram>` — chains `extractCluesFromPage` into `solveNonogramFromClues`.
**Depends on:** `modules/nonogram-domain.md` (`createNonogram`, `computeNonogramClues`'s `NonogramClues` type)

Notes:
- OCR accuracy on very small clue-number glyphs is imperfect even after upscaling (a few digits per puzzle can still be misread) — this is an accepted limitation, not a bug: `solveNonogramFromClues`'s strict all-or-nothing validation means misread clues overwhelmingly surface as a clear rejection (impossible or unsolvable-by-logic error) rather than a silently wrong puzzle, since a full-grid solution needs every line's clue to be simultaneously consistent. The URL/headless-browser path reads clue text directly and does not have this problem (confirmed accurate against the same real puzzle the OCR path previously misread).
- The URL import path is a personal-use decision made with the CGU conflict explicitly acknowledged, not resolved — see ADR `011` and `.vibe/backlog/done/020-import-nonograms-from-nonograms-org.md`.
- A Puppeteer-based render can take close to (occasionally over) 30s on a cold Chromium start; the web server's connection timeout was raised from 30s to 60s to accommodate this (see [[web-server]]).
