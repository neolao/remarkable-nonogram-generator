# Module: web-server
**Role:** Fastify HTTP server exposing the API and serving the static frontend. Exposes the reMarkable connection and send endpoints, a version check, the full `/api/nonograms/*` CRUD route set (list, create, fetch, update, delete), stateless SVG preview / PDF generation endpoints, a home page that lists/creates/deletes saved nonograms, and a manual grid editor page with a live row/column clue display.
**Files:** `packages/web/src/server.ts`, `packages/web/src/nonogram-routes.ts`, `packages/web/src/nonogram-client-clues.ts`, `packages/web/public/index.html`, `packages/web/public/app.js`, `packages/web/public/editor.html`, `packages/web/public/editor.js`, `packages/web/public/style.css`
**Exports:**
- `buildServer(options?): FastifyInstance` â€” registers static file serving, `GET /api/version`, the reMarkable routes ([[remarkable-cloud]], now also given the nonogram store so it can serve `POST /api/nonograms/:id/send`), and the nonogram routes; `options.nonogramsPath` overrides the default storage directory ([[nonogram-persistence]])
- `DEFAULT_PORT`, `resolvePort(env?)` â€” the server listens on `PORT` if set (validated as an integer in `1..65535`), else `DEFAULT_PORT`
- `registerNonogramRoutes(app, store)` â€” wires `GET /api/nonograms` to `NonogramStore.list()` (lightweight summaries, no cell data); `POST /api/nonograms` validates the grid via `createNonogram` and creates it (201) or returns 400 with a clear error; `POST /api/nonograms/preview` validates a grid the same way and returns its rendered SVG (200, `image/svg+xml`) via `renderNonogramToSvg` ([[nonogram-rendering]]) without touching storage, or 400 with a clear error; `POST /api/nonograms/generate` validates a grid the same way and returns its rendered PDF (200, `application/pdf`) via `renderNonogramToPdf` ([[nonogram-rendering]]) without touching storage, or 400 with a clear error; `GET /api/nonograms/:id` returns the full saved grid (200) or 404; `PUT /api/nonograms/:id` validates and updates an existing nonogram (200) or 404 if the id is unknown; `DELETE /api/nonograms/:id` removes it (204) or returns 404 if the id is unknown
- `initNonogramList()` (in `app.js`) â€” fetches `GET /api/nonograms` on the home page and renders each nonogram's name and size as a link to `editor.html?id=<id>`; shows an empty-state message when there are none, a "New nonogram" link to `editor.html` with no id, and a per-item delete button calling `DELETE /api/nonograms/:id` that removes the item from the DOM on success or shows an error and leaves it in place on failure
- `initEditor()` (in `editor.js`) â€” reads an optional `id` query parameter. With an id, fetches `GET /api/nonograms/:id` and renders its saved grid directly as an interactive cell grid (see [[decisions/005-minimal-editor-stub-page]] for the earlier placeholder this replaced), or shows the existing "not found" message on failure. Without an id, shows a width/height form; submitting it sends a blank grid of that size to `POST /api/nonograms/preview` purely to reuse its `createNonogram` validation (see [[decisions/006-editor-size-validation-via-preview-endpoint]]) â€” a 400 response's error message is shown as-is, a 200 response renders the blank interactive grid. Clicking a cell toggles it between filled and empty entirely client-side (in-memory grid state, no fetch per click); saving the edited grid back to the server is a separate not-yet-built feature. Every render and toggle also recomputes and displays row/column clues next to the grid, entirely client-side
- `computeClientNonogramClues(cells)` (`packages/web/src/nonogram-client-clues.ts`, tested) â€” computes row/column clues for a plain `boolean[][]` grid, matching [[nonogram-domain]]'s `computeNonogramClues`; hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) since the static page has no build step to import it
**Depends on:** `modules/remarkable-cloud.md`, `modules/nonogram-persistence.md`, `modules/nonogram-domain.md` (for `createNonogram` validation and the clue algorithm mirrored client-side), `modules/nonogram-rendering.md` (for `renderNonogramToSvg` and `renderNonogramToPdf`), `@remarkable-nonogram-generator/core` (for `CORE_VERSION`)
