# Module: web-server
**Role:** Fastify HTTP server exposing the API and serving the static frontend. Exposes the reMarkable connection and send endpoints, a version check, the full `/api/nonograms/*` CRUD route set (list, create, fetch, update, delete), stateless SVG preview / PDF generation endpoints, a home page that lists/creates/deletes saved nonograms, and a manual grid editor page with a live row/column clue display, a name/save flow that creates or updates the drawing, and a preview/download/send-to-reMarkable flow.
**Files:** `packages/web/src/server.ts`, `packages/web/src/nonogram-routes.ts`, `packages/web/src/nonogram-client-clues.ts`, `packages/web/src/nonogram-save-request.ts`, `packages/web/src/nonogram-send-request.ts`, `packages/web/public/index.html`, `packages/web/public/app.js`, `packages/web/public/editor.html`, `packages/web/public/editor.js`, `packages/web/public/style.css`
**Exports:**
- `buildServer(options?): FastifyInstance` — registers static file serving, `GET /api/version`, the reMarkable routes ([[remarkable-cloud]], now also given the nonogram store so it can serve `POST /api/nonograms/:id/send`), and the nonogram routes; `options.nonogramsPath` overrides the default storage directory ([[nonogram-persistence]])
- `DEFAULT_PORT`, `resolvePort(env?)` — the server listens on `PORT` if set (validated as an integer in `1..65535`), else `DEFAULT_PORT`
- `registerNonogramRoutes(app, store)` — wires `GET /api/nonograms` to `NonogramStore.list()` (lightweight summaries, no cell data); `POST /api/nonograms` validates the grid via `createNonogram` and creates it (201) or returns 400 with a clear error; `POST /api/nonograms/preview` validates a grid the same way and returns its rendered SVG (200, `image/svg+xml`) via `renderNonogramToSvg` ([[nonogram-rendering]]) without touching storage, or 400 with a clear error; `POST /api/nonograms/generate` validates a grid the same way and returns its rendered PDF (200, `application/pdf`) via `renderNonogramToPdf` ([[nonogram-rendering]]) without touching storage, or 400 with a clear error; `GET /api/nonograms/:id` returns the full saved grid (200) or 404; `PUT /api/nonograms/:id` validates and updates an existing nonogram (200) or 404 if the id is unknown; `DELETE /api/nonograms/:id` removes it (204) or returns 404 if the id is unknown
- `initNonogramList()` (in `app.js`) — fetches `GET /api/nonograms` on the home page and renders each nonogram's name and size as a link to `editor.html?id=<id>`; shows an empty-state message when there are none, a "New nonogram" link to `editor.html` with no id, and a per-item delete button calling `DELETE /api/nonograms/:id` that removes the item from the DOM on success or shows an error and leaves it in place on failure
- `initEditor()` (in `editor.js`) — reads an optional `id` query parameter, kept as a mutable `currentId` for the lifetime of the page. With an id, fetches `GET /api/nonograms/:id` and renders its saved grid directly as an interactive cell grid, pre-filling the name field (see [[decisions/005-minimal-editor-stub-page]] for the earlier placeholder this replaced), or shows the existing "not found" message on failure. Without an id, shows a width/height form; submitting it sends a blank grid of that size to `POST /api/nonograms/preview` purely to reuse its `createNonogram` validation (see [[decisions/006-editor-size-validation-via-preview-endpoint]]) — a 400 response's error message is shown as-is, a 200 response renders the blank interactive grid. Clicking a cell toggles it between filled and empty entirely client-side (in-memory grid state, no fetch per click). Every render and toggle also recomputes and displays row/column clues next to the grid, entirely client-side, and refreshes an SVG preview image and a PDF download via `POST /api/nonograms/preview` / `POST /api/nonograms/generate`. Clicking "Save" builds a save request from the name field and the current in-memory grid; on success the returned id is stored as `currentId` so a later save on the same page updates rather than duplicates. Clicking "Send to reMarkable" (with an optional folder field) is blocked with a message if the drawing has no `currentId` yet (see [[decisions/007-block-send-until-saved]]); otherwise it posts to `POST /api/nonograms/:id/send`, revealing the same pairing UI as the home page (posting to `/api/remarkable/pair`) when the response is the `not_authenticated` 409, and retrying the send once pairing succeeds
- `computeClientNonogramClues(cells)` (`packages/web/src/nonogram-client-clues.ts`, tested) — computes row/column clues for a plain `boolean[][]` grid, matching [[nonogram-domain]]'s `computeNonogramClues`; hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) since the static page has no build step to import it
- `buildNonogramSaveRequest(currentId, name, nonogram)` (`packages/web/src/nonogram-save-request.ts`, tested) — pure function deciding how to save a grid: rejects a blank/whitespace-only name with `{ ok: false, error }`, otherwise trims the name and returns `{ ok: true, request }` targeting `POST /api/nonograms` when `currentId` is null/absent or `PUT /api/nonograms/:id` when present; hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) for the same no-build-step reason
- `buildNonogramSendRequest(currentId, folder)` (`packages/web/src/nonogram-send-request.ts`, tested) — pure function deciding how to send a drawing to reMarkable: rejects a missing `currentId` with `{ ok: false, error }`, otherwise returns `{ ok: true, request }` targeting `POST /api/nonograms/:id/send` with the trimmed `folder` in the body, or omitted entirely when blank; hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) for the same no-build-step reason
**Depends on:** `modules/remarkable-cloud.md`, `modules/nonogram-persistence.md`, `modules/nonogram-domain.md` (for `createNonogram` validation and the clue algorithm mirrored client-side), `modules/nonogram-rendering.md` (for `renderNonogramToSvg` and `renderNonogramToPdf`), `@remarkable-nonogram-generator/core` (for `CORE_VERSION`)
