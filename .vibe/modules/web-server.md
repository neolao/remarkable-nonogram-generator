# Module: web-server
**Role:** Fastify HTTP server exposing the API and serving the static frontend. Exposes the reMarkable connection and send endpoints, a version check, the full `/api/nonograms/*` CRUD route set (list, create, fetch, update, delete) plus a per-id SVG preview endpoint and a per-id JSON export endpoint, stateless SVG preview / PDF generation endpoints, a nonograms.org URL-import endpoint and a JSON-file import endpoint, a home page that lists/creates/deletes/exports saved nonograms (each with a small thumbnail of its drawn grid) and links to the import page, a manual grid editor page with a live row/column clue display, a name/save flow that creates or updates the drawing, a preview/download/send-to-reMarkable flow (with an optional include-solution-page-2 toggle), a browser-side memory of the last-used reMarkable destination folder, and an import page with a puzzle-URL form (showing a progress indicator while the potentially slow import request is in flight) plus a second JSON-file import form. The frontend's CSS is mobile-first (see [[decisions/015-mobile-first-redesign-with-scrollable-pinned-grid]]): base rules target phone screens (44px-minimum touch targets, single-column flow), `min-width` media queries progressively adapt tablet/desktop layouts (the saved-nonogram list becomes a multi-column card grid, paired sections use a reusable `.layout-columns` side-by-side wrapper), and the grid editor's large-grid clue strips (`.column-clues`/`.row-clues`) use `position: sticky` so they stay pinned in view while the grid itself scrolls instead of shrinking its cells.
**Files:** `packages/web/src/server.ts`, `packages/web/src/nonogram-routes.ts`, `packages/web/src/nonogram-import-url-routes.ts`, `packages/web/src/nonogram-import-json-routes.ts`, `packages/web/src/nonogram-url-import-request.ts`, `packages/web/src/nonogram-json-import-file.ts`, `packages/web/src/nonogram-import-progress.ts`, `packages/web/src/nonogram-client-clues.ts`, `packages/web/src/nonogram-save-request.ts`, `packages/web/src/nonogram-send-request.ts`, `packages/web/src/remarkable-folder-cookie.ts`, `packages/web/public/index.html`, `packages/web/public/app.js`, `packages/web/public/editor.html`, `packages/web/public/editor.js`, `packages/web/public/import.html`, `packages/web/public/import.js`, `packages/web/public/remarkable-pairing.js`, `packages/web/public/style.css`
**Exports:**
- `buildServer(options?): FastifyInstance` â€” registers static file serving, `GET /api/version`, the reMarkable routes ([[remarkable-cloud]], now also given the nonogram store so it can serve `POST /api/nonograms/:id/send`), the nonogram routes, the nonogram URL-import routes, and the nonogram JSON-import routes; `options.nonogramsPath` overrides the default storage directory ([[nonogram-persistence]]); `options.importNonogramFromUrlFn` overrides the import pipeline (defaults to [[nonogram-url-import]]'s `importNonogramFromUrl`), letting tests substitute a fake instead of exercising a real headless browser; the Fastify instance sets a 60s `connectionTimeout` (raised from 30s to comfortably cover a headless-browser puzzle import on a cold Chromium start) and 5s `keepAliveTimeout` so a hung request doesn't hold a connection open indefinitely
- `DEFAULT_PORT`, `resolvePort(env?)` â€” the server listens on `PORT` if set (validated as an integer in `1..65535`), else `DEFAULT_PORT`; outside tests, a failed `app.listen()` (e.g. port already in use) is logged and exits the process with code 1 instead of an unhandled rejection
- `registerNonogramRoutes(app, store)` â€” wires `GET /api/nonograms` to `NonogramStore.list()` (lightweight summaries, no cell data); `POST /api/nonograms` validates the grid via `createNonogram` and creates it (201) or returns 400 with a clear error; `POST /api/nonograms/preview` validates a grid the same way and returns its rendered SVG (200, `image/svg+xml`) via `renderNonogramToSvg` ([[nonogram-rendering]]) without touching storage, or 400 with a clear error; `POST /api/nonograms/generate` validates a grid the same way and returns its rendered PDF (200, `application/pdf`) via `renderNonogramToPdf` ([[nonogram-rendering]]), forwarding an optional `includeSolution` body flag to add the solution page, without touching storage, or 400 with a clear error; `GET /api/nonograms/:id` returns the full saved grid (200) or 404; `PUT /api/nonograms/:id` validates and updates an existing nonogram (200) or 404 if the id is unknown; `DELETE /api/nonograms/:id` removes it (204) or returns 404 if the id is unknown; `GET /api/nonograms/:id/preview` loads a saved nonogram by id and returns its rendered SVG (200, `image/svg+xml`) via `renderNonogramToSvg` on its actual (drawn/solved) cells, or 404 if unknown â€” backs the home page's list thumbnails; `GET /api/nonograms/:id/export` loads a saved nonogram by id and returns it (200, `application/json`) as a [[nonogram-persistence]] `NonogramExport` via `serializeNonogramExport`, with a `Content-Disposition: attachment; filename="<slugified-name>.json"` header so the browser downloads it, or 404 if unknown
- `registerNonogramImportJsonRoutes(app, store)` (`nonogram-import-json-routes.ts`) â€” wires `POST /api/nonograms/import-json`: parses the raw request body (a malformed-JSON body is already rejected with a 400 by Fastify's own JSON content-type parser before the handler runs) via [[nonogram-persistence]]'s `parseNonogramImport`, saving the result (201) or returning 400 with the parser's own clear error (missing/non-array `cells`, invalid dimensions, oversized grid, etc.) â€” mirrors the dependency-injection-free shape of the URL-import route since there's no external pipeline to fake in tests, just the trusted `parseNonogramImport` core function
- `initNonogramList()` (in `app.js`) â€” fetches `GET /api/nonograms` on the home page and renders each nonogram as a link to `editor.html?id=<id>` containing a small thumbnail `<img>` (`GET /api/nonograms/:id/preview`), its name, and its size, followed by an actions group with an "Export" link (plain `<a href="/api/nonograms/:id/export" download>`, no JS needed since the server's `Content-Disposition` header drives the download) and a per-item delete button calling `DELETE /api/nonograms/:id` that removes the item from the DOM on success or shows an error and leaves it in place on failure; shows an empty-state message when there are none and a "New nonogram" link to `editor.html` with no id; every `fetch()` call in this file (list, delete, pairing status) is wrapped in try/catch, showing a "could not reach the server" message instead of an unhandled rejection on a network failure
- `initEditor()` (in `editor.js`) â€” reads an optional `id` query parameter, kept in a mutable `editorState.currentId` for the lifetime of the page. The function itself now only queries DOM elements (`queryEditorElements()`), builds initial state, wires event listeners, and picks the load-existing vs. new-grid bootstrap path â€” each flow (`renderClues`, `updatePreviewAndDownload`, `renderGrid`, `showGrid`, `handleSave`, `sendToRemarkable`, `loadExisting`, `handleSizeSubmit`) is a separate top-level function taking `elements`/state objects as explicit parameters, and every one of them wraps its `fetch()` call(s) in try/catch to show a "could not reach the server" message instead of leaving the UI stuck on network failure. With an id, `loadExisting` fetches `GET /api/nonograms/:id` and renders the saved grid directly as an interactive cell grid, pre-filling the name field (see [[decisions/005-minimal-editor-stub-page]] for the earlier placeholder this replaced), or shows the existing "not found" message on failure. Without an id, shows a width/height form; `handleSizeSubmit` sends a blank grid of that size to `POST /api/nonograms/preview` purely to reuse its `createNonogram` validation (see [[decisions/006-editor-size-validation-via-preview-endpoint]]) â€” a 400 response's error message is shown as-is, a 200 response renders the blank interactive grid. Clicking a cell toggles it between filled and empty entirely client-side (in-memory grid state, no fetch per click). Every render and toggle also recomputes and displays row/column clues next to the grid, entirely client-side, and refreshes an SVG preview image and a PDF download via `POST /api/nonograms/preview` / `POST /api/nonograms/generate`. Clicking "Save" (`handleSave`) builds a save request from the name field and the current in-memory grid; on success the returned id is stored as `editorState.currentId` so a later save on the same page updates rather than duplicates. On load, the reMarkable folder field is pre-filled from the `remarkable-folder` cookie via `readRemarkableFolderCookie`. An "Include solution on page 2" checkbox (unchecked by default) is read on every preview/download refresh and on send, re-triggering the preview/download refresh when toggled. Clicking "Send to reMarkable" (`sendToRemarkable`, with an optional folder field) is blocked with a message if the drawing has no `currentId` yet (see [[decisions/007-block-send-until-saved]]); otherwise it writes the folder field to the `remarkable-folder` cookie via `buildRemarkableFolderCookieAssignment` (a year-long cookie, or cleared if the field is blank) before posting to `POST /api/nonograms/:id/send`, revealing the same pairing UI as the home page when the response is the `not_authenticated` 409, and retrying the send once pairing succeeds
- `submitPairingCode(pairingCodeInput, pairingError, onSuccess)` (`packages/web/public/remarkable-pairing.js`) â€” the actual, single shared implementation (an ES module both `app.js` and `editor.js` `import`) of the reMarkable pairing-code submit flow: validates the code isn't blank, posts to `POST /api/remarkable/pair`, shows the error on failure (including on a network failure, guarded by try/catch), and on success clears the input and awaits the caller-supplied `onSuccess` (`showConnected` on the home page, `sendToRemarkable` on the editor page). Unlike the mirrored-TS-source pattern below, this logic had no TS-tested counterpart, so it was extracted into a genuinely shared browser module instead of duplicated.
- `buildRemarkableFolderCookieAssignment(folder)` / `readRemarkableFolderCookie(cookieHeader)` (`packages/web/src/remarkable-folder-cookie.ts`, tested) â€” pure functions for the `remarkable-folder` cookie: the former builds the `document.cookie` assignment string for a trimmed, percent-encoded folder value (or a `max-age=0` assignment that clears the cookie when the folder is blank), the latter extracts and decodes the folder value from a raw cookie header, returning `""` if absent or malformed; hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) for the same no-build-step reason
- `computeClientNonogramClues(cells)` (`packages/web/src/nonogram-client-clues.ts`, tested) â€” computes row/column clues for a plain `boolean[][]` grid by delegating to [[nonogram-domain]]'s `computeNonogramClues` (no longer reimplements the run-length algorithm itself); hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module, which itself mirrors `packages/core/src/nonogram-clues.ts`) since the static page has no build step to import it
- `buildNonogramSaveRequest(currentId, name, nonogram)` (`packages/web/src/nonogram-save-request.ts`, tested) â€” pure function deciding how to save a grid: rejects a blank/whitespace-only name with `{ ok: false, error }`, otherwise trims the name and returns `{ ok: true, request }` targeting `POST /api/nonograms` when `currentId` is null/absent or `PUT /api/nonograms/:id` when present; hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) for the same no-build-step reason
- `buildNonogramSendRequest(currentId, folder, includeSolution?)` (`packages/web/src/nonogram-send-request.ts`, tested) â€” pure function deciding how to send a drawing to reMarkable: rejects a missing `currentId` with `{ ok: false, error }`, otherwise returns `{ ok: true, request }` targeting `POST /api/nonograms/:id/send` with the trimmed `folder` in the body (omitted when blank) and `includeSolution: true` in the body when the parameter is truthy (omitted otherwise, default `false`); hand-duplicated as plain JS directly in `editor.js` (commented as a mirror of this tested module) for the same no-build-step reason
- A dedicated test (`packages/web/src/editor-mirror-sync.test.ts`) extracts the hand-mirrored function/constant source directly out of `editor.js` by name (brace-counting, no DOM/module import needed) and runs it against representative inputs, asserting the output matches the corresponding tested TS module â€” a sync guard against silent drift between the two copies, since nothing else enforces it
- `registerNonogramImportUrlRoutes(app, store, importFn?)` â€” wires `POST /api/nonograms/import-url` (JSON body: `{ url }`): returns 400 with a clear error if `url` is missing, otherwise calls `importFn` (the injected/default [[nonogram-url-import]] `importNonogramFromUrl`) and either saves the resulting nonogram (201, empty name the user can set later) or returns 400 with the pipeline's own error message (not-a-nonograms.org-URL, page unreachable/not found, or "could not fully solve"); `importFn` is injected with a real default per the project's established dependency-injection pattern, so route tests exercise real HTTP/persistence wiring against a fake pipeline instead of needing a real headless browser
- `buildNonogramUrlImportRequest(puzzleUrl)` (`packages/web/src/nonogram-url-import-request.ts`, tested) â€” pure function validating the URL-import form before submit: rejects a blank url or one that fails `new URL(...)` parsing with `{ ok: false, error }`, otherwise returns `{ ok: true, request }` targeting `POST /api/nonograms/import-url` with the trimmed url; hand-duplicated as plain JS directly in `import.js` (commented as a mirror, with its own `nonogram-url-import-mirror-sync.test.ts` sync guard) for the same no-build-step reason
- `setImportInProgress(toggle: { submitButton: { disabled }, progressElement: { hidden } }, inProgress)` (`packages/web/src/nonogram-import-progress.ts`, tested) â€” pure toggle function used to show/hide a "please wait" indicator: sets `submitButton.disabled` and `progressElement.hidden` from a single `inProgress` boolean; hand-duplicated as plain JS directly in `import.js` (commented as a mirror of this tested module, with its own `nonogram-import-progress-mirror-sync.test.ts` sync guard) for the same no-build-step reason
- `initImportUrlForm()` (in `import.js`) â€” wires the import page's URL form: on submit, validates via the mirrored `buildNonogramUrlImportRequest`, posts the url as JSON to `POST /api/nonograms/import-url`; while that request is in flight, shows a spinner + "Import in progress..." message and disables the Import button via `setImportInProgress` (since the URL-import pipeline runs headless-browser rendering server-side and can take up to a minute, with no intermediate progress data available â€” see [[nonogram-url-import]]), hiding it again on both the business-error and network-failure branches; on success navigates to `editor.html?id=<id>`, on a 400 or network failure shows the server's error message (or a "could not reach the server" fallback) in place without navigating away
- `validateNonogramJsonImportFile(file)` (`packages/web/src/nonogram-json-import-file.ts`, tested) â€” pure function validating the JSON-file import form before submit: rejects a null/undefined file selection with `{ ok: false, error: "Please choose a JSON file to import" }`, otherwise `{ ok: true }`; hand-duplicated as plain JS directly in `import.js` (commented as a mirror, with its own `nonogram-json-import-file-mirror-sync.test.ts` sync guard) for the same no-build-step reason
- `initImportJsonForm()` (in `import.js`) â€” wires the import page's second, JSON-file form: on submit, validates a file was chosen via the mirrored `validateNonogramJsonImportFile`, reads it client-side with `file.text()`, and forwards that raw text as-is (no re-parse/re-stringify round trip) as the body of `POST /api/nonograms/import-json` with an `application/json` content type, relying on the server (Fastify's body parser, then [[nonogram-persistence]]'s `parseNonogramImport`) for all validation; shows a brief "Import in progress..." indicator via the same `setImportInProgress` toggle used by the URL form; on a non-2xx response prefers the response body's `message` field over `error` (Fastify's own malformed-JSON error puts the useful detail in `message` and only the generic "Bad Request" in `error`, while this route's own validation errors only ever set `error`), falling back to a generic message; on success navigates to `editor.html?id=<id>` like the URL-import form does
