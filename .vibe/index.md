## Modules
- [`modules/docker-deployment.md`](.vibe/modules/docker-deployment.md) — packages the app as a single Docker image (multi-stage build, system Chromium for multi-arch support, volume-persisted config directory), with a compose example and a Synology Container Manager deployment guide
- [`modules/remarkable-cloud.md`](.vibe/modules/remarkable-cloud.md) — reMarkable Cloud pairing, credential storage, PDF upload, and the `sendNonogramToRemarkable` use case (core layer generic/domain-level); the web routes layer is a thin HTTP mapper on top of it
- [`modules/nonogram-domain.md`](.vibe/modules/nonogram-domain.md) — core nonogram grid data model (filled/empty cells) and row/column clue computation, validated against reMarkable 2 page size constants
- [`modules/nonogram-rendering.md`](.vibe/modules/nonogram-rendering.md) — renders a nonogram grid and its clues to SVG (live preview) and to a reMarkable-2-sized PDF (download/send)
- [`modules/nonogram-persistence.md`](.vibe/modules/nonogram-persistence.md) — file-based save/list/load/delete of nonograms, one JSON file per id, backs the full `/api/nonograms` list/create/get/update/delete routes; also the JSON export/import transfer format for backing up or moving a single saved nonogram
- [`modules/nonogram-url-import.md`](.vibe/modules/nonogram-url-import.md) — reconstructs a nonogram from a nonograms.org puzzle URL rendered via headless browser (clues read directly as page text), running the shared logic-only nonogram solver rather than trusting the source's cells; the previous OCR-based image-upload import path was removed (backlog item 024) once this path proved more accurate
- [`modules/web-server.md`](.vibe/modules/web-server.md) — Fastify server + static frontend (mobile-first responsive design, see ADR 015), exposes the reMarkable connection/send flow, the full nonogram list/create/get/update/delete API plus a per-id SVG preview endpoint and a per-id JSON export endpoint, stateless SVG preview / PDF generation endpoints, a nonograms.org URL-import endpoint and a JSON-file import endpoint, a home page listing/creating/deleting/exporting saved nonograms (each with a small grid thumbnail, laid out as a multi-column card grid on wide screens) and linking to the import page, a manual grid editor page with a live client-side row/column clue display and a scrollable grid with edge-pinned clues on large puzzles, a name/save flow (create then update, no duplicates), a live preview/PDF-download/send-to-reMarkable flow (configurable listen port via `PORT`), a browser cookie remembering the last-used reMarkable destination folder across visits, and an import page with a puzzle-URL form (showing a progress indicator while the import request is in flight) plus a JSON-file import form

## Observed patterns
- npm workspaces monorepo (`packages/core`, `packages/web`); `core` has no Node-specific/browser-specific runtime dependency baked into its public API, so it stays a pure, testable domain layer
- `packages/core/src/` is split into three DDD layers — `domain/` (pure entities/value objects and domain services: `Nonogram`, `NonogramClues`, grid validation, clue computation, line solver, JSON transfer shape — zero dependency on `application/` or `infrastructure/`), `application/` (use cases and the ports they depend on: `sendNonogramToRemarkable`, `importNonogramFromUrl`, the `NonogramStore`/`CredentialStore` interfaces), and `infrastructure/` (technical adapters: SVG/PDF rendering, the reMarkable Cloud client, Puppeteer-based page scraping, the network-timeout helper); `packages/core/src/index.ts` re-exports the same public names regardless of layer, so `packages/web` (which only ever imports via the `@remarkable-nonogram-generator/core` package alias) needed no changes (see ADR 016)
- TypeScript in `strict` mode, ES modules (NodeNext), each package compiles independently via its own `tsconfig.json` extending the root `tsconfig.base.json`
- Tests co-located with source (`*.test.ts` next to the file it covers), run via Vitest from the repo root (`vitest.config.ts` globs all packages)
- Style and formatting enforced by Biome (`npm run lint`)
- Fastify routes are grouped by concern into small `register*Routes(app, ...)` functions (see `remarkable-routes.ts`) rather than one monolithic route file
- Route handlers stay thin HTTP mappers over a domain-level use-case function (e.g. `sendNonogramToRemarkable`) that returns a discriminated-union result — keeps the outcome logic unit-testable without Fastify and independently reusable
- Frontend is plain static HTML/CSS/JS, loaded as native ES modules (`<script type="module">`, no build step) served directly by Fastify's `@fastify/static`; genuinely shared browser-only logic with no TS-tested counterpart (e.g. the pairing-code submit flow) lives in its own `public/*.js` module and is `import`ed by the pages that need it, rather than duplicated
- Logic needed both server-side and client-side (e.g. clue computation, save/send request building) is written once as a tested TS module under `packages/web/src/`, then hand-duplicated as plain JS directly into the relevant `public/*.js` file with a comment noting the mirrored source, since the static frontend has no build step to import it; a dedicated test (`editor-mirror-sync.test.ts`) extracts the mirrored source by name and checks its behavior against the tested TS module to guard against silent drift
- Web-layer file-backed implementations of a core-defined store interface are named `file-<thing>-store.ts` (e.g. `file-nonogram-store.ts`, `file-credential-store.ts`), distinct from the core package's same-concept interface file (`application/nonogram-store.ts`, `application/remarkable-credential-store.ts`), to avoid identically-named interface/implementation modules across packages
- External clients with no timeout/abort option of their own (`rmapi-js`) are wrapped in a small `withTimeout(promise, ms, message)` race helper rather than left to hang indefinitely
- Side-effecting dependencies on a third-party client are injected with a real default (e.g. `authenticate`'s `client: RemarkableClient = { register, remarkable }`), so tests can substitute a fake without mocking the module
- Puppeteer `page.evaluate()` callbacks must avoid named/assigned inner functions (`function foo() {}` or `const foo = () => {}`) — esbuild's `__name()`-keeping transform (active under `tsx`/Vite/Vitest) injects a call to a helper that doesn't exist once the callback is serialized and re-run inside the browser's isolated context, throwing `__name is not defined` at runtime; this only surfaces when the code actually executes a real browser (unit tests using a fake `PageRenderer` won't catch it) — inline the logic instead
- Any `public/style.css` selector that sets `display` on an element also toggled via the `hidden` attribute/property (e.g. `.import-progress`) needs an explicit `.selector[hidden] { display: none; }` override — an author-stylesheet `display` rule always wins over the browser's default `[hidden] { display: none }` UA rule regardless of source order, so without the override the element stays visible even while `hidden` is set (a real bug hit and fixed on `.import-progress`, guarded by a content-based test reading `style.css` since this project has no jsdom/browser-rendering test environment)
- `packages/web`'s `start` script (and therefore the Docker image, [[docker-deployment]]) runs the server via `tsx` on TypeScript source, the same mechanism as local dev (`tsx watch`) — not `node` on the `tsc`-compiled `dist` output, since `core`'s `package.json` deliberately exports its own TypeScript source rather than a built `dist` (see ADR 013)
- `public/style.css` is mobile-first (see ADR 015): unprefixed/base rules target phone screens directly, `@media (min-width: ...)` blocks progressively adapt tablet/desktop rather than the reverse (`max-width` queries stripping down a desktop layout); a reusable `.layout-columns` wrapper class stacks its children on narrow screens and becomes a 2-column CSS grid at `min-width: 900px` for any page section with two paired cards (used by both the editor and the import page); genuinely falsifiable CSS behavior (sticky-positioned pinned elements, minimum touch-target sizing, a specific breakpoint switching a layout to `display: grid`) is covered by a content-based test reading `style.css`/the HTML files directly (`mobile-responsive-style.test.ts`), following the same no-jsdom pattern as `import-progress-style.test.ts`; full visual correctness is verified by loading the real pages in a headless browser at multiple viewport widths rather than asserted in these tests
